# Project Charter

## Vision

Project Evergreen exists to create a Christmas experience that unfolds over six months instead of a single day.

The project serves two purposes:

1. Deliver an unforgettable Christmas surprise for Dad.
2. Document the complete engineering journey from idea to completion.

---

# Mission Statement

Build something that is meaningful, beautiful, maintainable, and worthy of sharing.

Every improvement should make the project stronger while preserving the story of how it was built.

---

# Engineering Standards

## Rule 1 — Never Rewrite History

History is never deleted.

Documentation is always appended.

Mistakes are documented rather than hidden.

---

## Rule 2 — Chronological Documentation

All documentation is maintained from oldest to newest.

Example:

Session 001

↓

Session 002

↓

Session 003

↓

Current Session

---

## Rule 3 — Every Coding Session Ends With Documentation

Before every release:

- Release Notes updated
- Engineering Notes updated
- Lessons Learned updated
- Roadmap reviewed
- Project History updated (when applicable)

Documentation is considered part of development.

---

## Rule 4 — One Feature at a Time

Every development task follows the same process:

Choose one feature.

Implement it.

Test it.

Document it.

Commit it.

Deploy it.

Small, tested improvements are preferred over large untested changes.

---

## Rule 5 — Single Source of Truth

Application data should exist in one place whenever possible.

The Project Object inside JavaScript is the authoritative source for dashboard information.

---

## Rule 6 — Meaningful Progress

The dashboard progress percentage only increases after meaningful engineering milestones.

Progress reflects project maturity, not hours worked.

---

## Rule 7 — Versioning

Every public release receives:

- Version Number
- Codename
- Release Notes

Example:

v0.5 — Brain

---

## Rule 8 — Preserve the Mystery

The dashboard should reveal progress without revealing the final gift.

Curiosity is part of the experience.

---

## Rule 9 — Professional Quality

Project Evergreen should be developed using professional software engineering practices whenever practical, including:

- Git version control
- Meaningful commits
- Documentation
- Roadmaps
- Engineering journals
- Release notes

---

## Rule 10 — Build With Purpose

Every feature should answer at least one question:

Does this improve the experience for Dad?

Does this improve the quality of the project?

Does this teach a new engineering concept?

If the answer is no, reconsider the feature.
